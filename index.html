Jellyfish Documentation

Jellyfish is a bare-bones (kinda like a Jellyfish!) mobile-first HTML/SCSS framework designed to create flexible grid-based layouts, with only the most essential elements and utilities built in. Jellyfish allows for a solid base for web development work, reducing the bloat and clutter of more commercial and well-known products. Jellyfish is designed for ease-of-use expansion and is a constantly evolving product ready to adapt to any web project.

Framework last updated: February 2019

Directory Structure

Jellyfish uses a fairly straightforward directory structure, outlined below. Partials are used to split the scss into manageable sections.

/assetsHolds all required assets.
/css/style.cssCore stylesheet as compiled by base.scss.
/js/scripts.cssBy default, only contains jQuery required by the navigation menu.
/scssHolds scss files. Files are listed below in the order that they are @included in base.scss
/base.scssCompiles all partials into ../css/style.css .
/_mixins.scss and /mixinsContains mixins to aid in the development of scss files.
/_settings.scssStores Jellyfish settings including grid settings, breakpoint settings, default font settings, color map and base sizes.
_reset.scssContains the fantastic CSS reset by Eric Meyer as well as setting all elements to box-sizing: border-box
/_global.scssContains scss that modifies elements found across all areas of a website, including typography, links, images and utilities.
/_grid.scssBuilds the grid system.
/_nav.scssContains scss that modifies elements found in the main navigation area of a website.
/header.scssContains scss that modifies elements found in the main header area of a website.
/_content.scssContains scss that modifies elements found in the main content area of a website.
/_sidebar.scssContains scss that modifies elements found in the sidebar / widgets area of a website.
/_footer.scssContains scss that modifies elements found in the main footer area of a website.
index.htmlDocumentation. Can be safely removed for live builds.

Basic HTML elements

Typography

Global font families, line height and size are defined in _settings.scss. With the grid breakpoint settings, a new $base-font-size and $base-line-height can be defined for each breakpoint. These settings are applied via a loop in the body selector in _global.scss.

// fonts
$font-primary: 'Nunito', 'Arial', sans-serif;
$font-secondary: 'Poppins', 'Arial', sans-serif;

// base sizes (used before breakpoints are applied)
$base-font-size: 1.6em;
$base-line-height: 1.5;

Hierarchy

All text will default to the global $base-font-size unless otherwise specified. Font size hierarchy is defined in _global.scss and can be implemented by extending the %font-size placeholder via @extend %font-size{$size}. This is predominantly used in Headings but can be used with any element.

// Font Sizes
%font-size{
    &--xsmall { font-size: 0.6em};
    &--small { font-size: 0.8em };
    &--normal { font-size: 1em};
    &--medium { font-size: 1.5em};
    &--large { font-size: 1.75em};
    &--xlarge { font-size: 2em};
    &--xxlarge { font-size: 2.5em};
    &--xxxlarge { font-size: 3em};
}

As defined in _global.scss, all img elements have height and width set to auto, and max-width set to 100%. This prevents images from breaking out of their containers, and also allows width/height to scale as appropriate when a more specific dimension is defined.

Lists

Support for basic lists is included for ul, ol and dl. Minimal styling is included, with lists styled via _global.scss. The placeholder class %lists is used to output consistent styling to both ul and ol lists:

Utilities

Hiding elements

Elements can be hidden via .hide to hide them on all devices, or by using breakpoint selectors to target specific screen sizes - see breakpoint settings. The grid system will automatically generate the classes as required, and name them according to the class names defined in the $breakpoints map in _settings.scss. An element can also be set to be hidden (but still take up space on the page) with the class .invisible. These css classes can be chained to hide/show elements at different breakpoints.

.invisible This text is invisible but still takes up space on the page. So if it falls across multiple lines, the table cell will grow to fit it.
.hide This text will be hidden on all devices.
.hide-below-sm This text will be hidden below the sm breakpoint.
.hide-above-sm This text will be hidden above the sm breakpoint.
.hide-below-md This text will be hidden below the md breakpoint.
.hide-above-md This text will be hidden above the md breakpoint.
.hide-below-lg This text will be hidden below the lg breakpoint.
.hide-above-lg This text will be hidden above the lg breakpoint.
.hide-below-sm .hide-above-lg This text will only be shown in the md breakpoint.

Grid

Grid Settings

At the heart of this framework is a grid that is completely customisable with the following settings:

  • Column quantity
  • Push/Pull include y/n
  • Offset include y/n
  • Debug mode on/off
  • Endless user-defined breakpoints with customisable names (Jellyfish does not force you to conform to any naming or sizing conventions) and settings:
    • Container width
    • Container gutter
    • Column gutter
    • Base font size
    • Base line height

Breakpoint settings

Jellyfish is designed to be flexible, allowing an infinite number of user-defined (and user-named) breakpoints - declared through the $breakpoints map in _settings.scss. When compiled, the css will automatically be created for any breakpoints defined in this map (.col-$breakpoint and utilities i.e. .push/.pull/.offset/.hide).

Example breakpoint code

sm: (
    breakpoint: 400px, // px
    container-width: 100%, // % or rem
    container-gutter: rem(10px), // rem
    col-gutter: rem(20px), // rem
    base-font-size: 1.4em, // em
    base-line-height: 1.48 // number, % or em
)

By default breakpoints are named sm, md and lg however you can name breakpoints as you wish to suit your project requirements and personal naming conventions (e.g. small, medium, large, gigantic, mammoth, microscope, mobile, tablet, laptop, desktop). Jellyfish will use these names to generate the code for columns and modifiers as detailed above.

Required Variables

For each breakpoint, the following variables must be defined (all others are optional - any absences will simply inherit settings from the previous breakpoint):

  • breakpoint
  • container-width
  • col-gutter

Terminology

breakpointThe width at which the breakpoint will take affect. This must be provided in px and will be converted to an em value through the breakpoint mixin. This variable is required.
container-widthThe width of the .container element at this breakpoint - should be provided as %, or rem. px can be converted to rem through the rem() mixin. This variable is required.
container-gutterPadding applied to the left and right of the .container element - this should be provided as rem value - px can be converted via the rem() mixin.
col-gutterPadding applied to the left and right of .col elements - this should be provided as a rem value - px can be converted via the rem() mixin. This variable is required.
base-font-sizeGlobal base font size (applied to body) which will affect all elements sized with em. Useful for improving legibility at different breakpoints. This must be provided as an em value and can be converted from px via em().
base-line-heightGlobal base line-height (applied to body) which will affect all elements that do not have a line-height applied. Can be provided as a number, percent, em or rem

Relevant code

body {
    @each $point, $i in $breakpoints {
    // set base font size and line height for each specified media query
   @include above($point) {
       font-size: bp-query($point, base-font-size);
       line-height: bp-query($point, base-line-height);
   }
}

@each $point,
$i in $breakpoints {
    // for each breakpoint defined...
    @include above($point) {
        // if a breakpoint is found, set media query min-width
        .container {
            // give our container a width/max width, and padding to border content
            width: bp-query($point, container-width);
            max-width: bp-query($point, container-max-width);
            padding-left: bp-query($point, container-gutter);
            padding-right: bp-query($point, container-gutter);
        }
    }
}

Column amount

The underlying benefit of Jellyfish is it's flexible approach to grid layout. You can define the precise amount of columns required, and on compile, the scss logic will produce all of the required css styling. So whether you want a 4 column, 7 column, 12 column or 3000 column grid (Not sure why you would want this, but knock yourself out...) Jellyfish has you covered. For the purposes of this documentation, we are using a 12 column grid - but feel free to go wild with a 5 column grid if your project requires it.

This ability to define only the required amount of columns allows the compiled code to be optimised and lean, reducing unnecessary bloat.

Column quantity is required and is defined by the variable $cols in _settings.scss and the logic for generating the grid is defined in _grid.scss.

Push / Pull

Columns can be pushed (to the right) or pulled (to the left) by a certain number of columns, if $has_push_pull is set to true in _settings.scss. .push and .pull classes are generated automatically by _grid.scss

Elements can be pushed/pulled at different breakpoints as defined in the breakpoint settings by using the classes .push-$breakpoint-$cols and .pull-$breakpoint-$cols respectively. Push and pull can also be reset at a breakpoint with the classes .push-$breakpoint-0 and .pull-$breakpoint-0 respectively.

.push and .pull classes can be chained eg. .push-sm-2 .push-md-5 .push-lg-0

.push-sm-6 .push-md-8 .push-lg-4
.pull-sm-6 .pull-md-0. pull-lg-4
.pull-md-8 .pull-lg-4

Relevant Code

//Found within the @each loop
@if $has_push_pull {
    $diff: 100% - ($one_col * $i);
    .push-#{$point}-#{$cols - $i} {
        left: $diff;
    }
    .pull-#{$point}-#{$cols - $i} {
        right: $diff;
    }
}

Offset

Similar to push / pull, an .offset can be applied to a column to move it (and any elements that follow it) to the right. Offset is enabled by setting $has_offset to true in _settings.scss

Where .push uses left positioning to move, this can cause columns to overlap. .offset avoids this by using margin, which forces neighbouring columns to move accordingly.

Elements can be offset at different breakpoints as defined in the breakpoint settings by using the class .offset-$breakpoint-$cols. Offset can also be reset at a breakpoint with the class .offset-$breakpoint-0. .offset can also be chained eg. .offset-sm-3 .offset-md-6 .offset-lg-0

Grid Structure

Jellyfish uses a simple grid structure, consisting of a .container, .row and several .col elements. These elements are defined in _grid.scss

.containerBy default, the grid will fill all available space (ie 100% width). To restrict the width of the grid and keep elements neatly aligned, a .container should be used. The container is centered and can be customised at each breakpoint with the following settings: width, max-width, padding (left/right). See grid breakpoint settings for information.
.rowHouses a number of .col elements and is automatically cleared with .clearfix to prevent flow issues. To preserve equal margins on the left and right, the only elements inside a .row should be .col elements
.colA single vertical column. Can be used with breakpoint based modifiers to change width at different breakpoints. With no modifiers, a .col element will be 100% width.

Valid examples

<div class="container">
    <div class="row">
        <div class="col">
            .col element inside a .container .row
        </div>
    </div>
</div>

Collapsing Rows/Columns

.row and .col elements can be collapsed (ie. left and right margin/padding set to 0) by applying the utility class .collapse as per the examples below. Collapsing elements will but their left and right edges up to the edge of the .container element. !important is used to avoid specificity issues.

When applied to a .row element, all child .col elements will be collapsed. When applied to a .col element, only that element will be collapsed.

Padding Rows

Padding can be added quickly to a .row element with the class .padded to create white space around rows. By default, 7rem padding is added to the top and bottom of the .row; this can be changed via _grid.scss.

.row.padded
.row {
    &-padded {
    padding-top: 7rem;
    padding-bottom: 7rem;
    }
}

Columns

A .row consists of several .col elements, which (usually) add up to the quantity determined by the column amount variable ($cols). These elements are the building blocks of any layout.

Too many / too few columns

Should the number of a row's columns total more than defined by $cols, excess .col elements will simply wrap onto the next line.

If the total number of used columns is not the same as the column amount defined by $cols (or a multiple thereof), the last .col element will appear floated to the right. This behaviour can be reset by adding the class .reset-float to the element, to float it to the left. This class can not currently target breakpoints.

Columns are floated in this way to force all browsers to align the right edges of rows neatly. As column widths are calculated with percentages, this can cause alignment issues on some browsers, should this styling not be applied.

Adjusting column width via breakpoints

A column's width can be altered at different breakpoints, by using breakpoint modifiers as determined by the grid breakpoint settings. For example, a single column can change from being full-width, to half-width, to quarter-width as the browser width increases. If no breakpoints are defined, or at the smallest browser width (before any breakpoint settings take effect), a .col element will span 100% width.

Modifiers are formatted as such: .$breakpoint-$cols and can be chained eg. .sm-12 .md-6 .lg-4

Jellyfish will automatically generate all required code for columns, based on the variables defined in _settings.scss ($cols and $breakpoints). The code that generates columns is defined within an @each loop found inside _grid.scss.

Nesting .row and .col elements

.row and .col elements can be infinitely nested within each other, so long as one golden rule is followed: All immediate child elements of a .row must be .col elements. A .col element can contain a .row but that must follow this same rule.

Note that because column widths are defined as percentages, when you nest a row, it's child columns will span XX% of the parent column's width. i.e. if you nest a row in a .col-sm6, adding a .col.sm-6 will effectively make that element a .col.sm-3 of the parent row.

Settings

Debug mode

Jellyfish comes with a handy debug mode that can be enabled in two ways:

  1. by setting $debug-mode-on to true in _settings.scss.
  2. by adding the class .jf-debug to the body element, or a parent element (either within the code itself or through a browser inspector).

Debug mode allows you to quickly see the structure of the grid, and diagnose any layout issues.

When debug mode is enabled, all .row elements will glow in pink and all .col elements will glow in turquoise.

Additionally, all .row elements will have their background striped to show the number of columns in the grid structure.

Providing that $has_push_pull and/or $has_offset are set to true in _settings.scss, any element affected by a .push, .pull or .offset at the active breakpoint will be highlighted with a red border; the side with the heavier stroke indicating the direction to which the action has occured.

Rem and Em conversion

The functions rem($value) and em($value) defined in mixins/_rem.scss can be used to convert pixel measurements into rem and em respectively. As pixels do not scale with font-size, rem and em should always be used to specify measurements.

Unit Based on Use for
Rem HTML document font size. By default this is set to 62.5%, meaning that all rem values are based on a font size of 10px, making rem values very easy to interpret - e.g. 4rem is 40px. See the mixin rem-baseline in mixins/_rem.scss for more information. Elements that should scale with the browser font settings with a view to be consistent across the site, regardless of parent font size. e.g. layout, margins, etc
Em Body font size or the font-size of the element/parent (whichever is the closest in the cascade). The variable $base-font-size in the breakpoint map can be used to change the font-size of the body element at different breakpoints, allowing good control over font size changes. Text hierarchy and related margins, padding, etc

The rem() function / mixin was written by Pierre Burel. For source code and documentation visit the GitHub repository

Examples

Rem can be used as both a mixin and function. All of the below uses are valid markup.

    @include rem(margin, 15px 0);
}
h1 {
    font-size: rem(20px);
}

    font-size: em(25px);

Global smooth() transition

Use the mixin @include smooth($property) to apply a global transition for a consistent UI feel. By default, the transition is 0.3s ease however this can be changed in _mixins.scss.

This mixin currently accepts only one property variable - eg. all, width, color, font-size, etc and uses Bourbon to produce all vendor prefixes. By default, Jellyfish uses this mixin on anchor tags and the navigation menu.

Examples

Anchor tags have smooth(color) applied by default - hover here to preview.

This element has smooth(transform) applied to it. Hover to preview.

This element has smooth(background) applied to it. Hover to preview.

Relevant Code
@mixin smooth($var) { // Global transition
    @include transition($var 0.3s ease);
}

Colour Map

Jellyfish makes use of a map for color values, to allow consistent and easy styling of elements. Colors are defined in the $colors map in _settings.scss and the function is defined in _mixins.scss. Colors can be applied to an property with the function color($color). The color() function is used throughout Jellyfish' codebase.

Example usage
$colors: (
    text: #403F41;
    primary: #cc0000;
);
body {
 color: color(text);
 background: color(primary);
}

Breakpoint Queries

Jellyfish comes with three breakpoint query mixins, allowing you to target styles to specific breakpoints. The codebase for these is defined in _mixins.scss. These mixins are flexible, allowing you to specify either a $breakpoint name (e.g. sm) or a px value (which is converted automatically by the em() mixin.

@include above($min) {...}CSS contained within the brackets will apply above the given breakpoint, until overwritten by specificity or a higher breakpoint.
@include between($min, $max) {...}CSS will apply between the two specified breakpoints only.
@include below($max) {...}CSS applies below the given breakpoint, unless overridden by specificity. As a general rule, it's best to use above() rather than below().
Example usage
body {
    @include above(sm) {
        background-color: blue;
    }
}
@include between(sm,md) {
    h1 {
        font-size: 3em;
    }
}
.element {
    @include below(600px) {
        display: block;
    }
    @include between(600px,lg) {
        display: inline-block;
    }
    @include above(lg) {
        display: none;
    }
}
Relevant Code
@function bp-query($point, $var) {
    // Function to query our breakpoint map
    // Accepts args $breakpoint-size, $variable
    @return map-get(map-get($breakpoints, $point), $var);
}

// Media queries check the breakpoint map to see if a given string exists, else they convert a given pixel value to em.
@mixin above($min) {
    $bp-min: $min;
    @if map-has-key($breakpoints, $min) {
        $bp-min: bp-query($min, breakpoint);
    }
    @warn "Unknown `#{$min}` in $breakpoints";
    @media (min-width: em($bp-min)) {
        @content;
    }
}
@mixin between($min, $max) {
    $bp-min: $min;
    @if map-has-key($breakpoints, $min) {
        $bp-min: bp-query($min, breakpoint);
    }
    @warn "Unknown `#{$min}` in $breakpoints";
    $bp-max: $max;
    @if map-has-key($breakpoints, $max) {
        $bp-max: bp-query($max, breakpoint);
    }
    @warn "Unknown `#{$max}` in $breakpoints";
    @media (min-width: em($bp-min)) and (max-width: em($bp-max - 1px)) {
        @content;
    }
}
@mixin below($max) {
    $bp-max: $max;
    @if map-has-key($breakpoints, $max) {
        $bp-max: bp-query($max, breakpoint);
    }
    @warn "Unknown `#{$max}` in $breakpoints";
    @media (max-width: em($bp-max - 1px)) {
        @content;
    }
}
