Welcome to Jellyfish.

Yet another lightweight SCSS framework to add to the ever-growing list.

Jellyfish has a few useful features that other frameworks don't have; primarily the ability to give breakpoint modifiers your own class names (want to name them micro, medium, mammoth - go for it! and the ability to define how many columns are in your grid (want a 13 column grid? You go for it, tiger! I won't judge).

Aside from that, Jellyfish has all the usual styling, classes and modifiers you have come to expect in a modern flex-based CSS framework.

Jellyfish is powered by Grunt.js to compile scss and javascript, adding prefixers, concatenating and minifying in the process.

HTML Boilerplate

Begin your project with as basic HTML5 boilerplate that does the following:

  • Sets the default language and character set.
  • Uses the latest Internet Explorer rendering mode.
  • Defines the width of the viewport to enable responsiveness.
  • Includes links to the minified CSS file and minified javascript file.

Grid

Jellyfish's grid system is built on flexbox. The grid consists of containers (.container), rows (.row) and columns (.col ). These elements are defined in /build/scss/jellyfish/components/_grid.scss.

A basic 2 column grid

Containers

Containers keep rows and columns - contained! By default, a row will fill all available space (ie 100% width). To restrict the width of the grid and keep elements neatly aligned, a .container can be used. The container is centered and can be customised at each breakpoint with the following settings: width, max-width, padding (left/right). See grid breakpoint settings for information. A container with an additional class of .full-width will span 100% of available space.

Rows

Rows contain a number of .col elements, grouped horizontally. To preserve equal margins on the left and right, the only elements inside a .row should be .col elements.

Columns

Unlike most grid systems, Jellyfish allows you customise how many columns your grid will use. This is determined by the $cols variable in /build/scss/_settings.scss.

Columns are equal-width by default, and expand to fill all available width.

Nested Grids

.row and .col elements can be infinitely nested within each other, so long as one golden rule is followed: A .row can only contain .col elements.

Note: because column widths are defined as percentages, when you nest a row, it's child columns will span XX% of the parent column's width. e.g. if you nest a row in a .col.sm-6, adding a .col.sm-6 will effectively make that element a .col.sm-3 of the parent row.

Breakpoint Settings

Jellyfish is designed to be flexible, allowing an infinite number of user-defined (and user-named) breakpoints - declared through the $breakpoints map in /build/scss/_settings.scss. Upon code compile, css class names will be generated for each breakpoint, for columns .$breakpoint-* and modifiers .order-$breakpoint-*, .hide-$breakpoint-*.

Example breakpoint definition

            
              sm: (
                breakpoint: 400px,
                container-width: 96%,
                container-gutter: 20px,
                col-gutter: 20px,
                root-font-size: 16px,
                root-line-height: 1.48em
                ),

By default, breakpoints are named sm, md, lg and xl however you can name breakpoints as you wish to suit your project requirements and personal naming conventions (e.g. microscopic, small, medium, large, gigantic, mammoth, mobile, tablet, laptop, desktop, jumbotron). Jellyfish will use these names to generate the code for css classes as detailed above.

If you choose not to declare certain values, they will be inherited.

Terminology

TermDescription
breakpointThe width at which the breakpoint will take effect. This must be provided in px and will be converted to an em value through the breakpoint mixin. This variable is required.
container-widthThe width of the .container element at this breakpoint - should be provided as %, px or rem. This variable is required.
container-gutterPadding applied to the left and right of the .container element - this should be provided as px value.
col-gutterPadding applied to the left and right of .col elements - this should be provided as a px value. This variable is required.
root-font-sizeGlobal base font size (applied to body) which will affect all elements sized with em. Useful for improving legibility at different breakpoints. This should be provided as a px value.
root-line-heightGlobal base line-height (applied to body) which will affect all elements that do not have a line-height applied. Can be provided as a number, em or px.

Note: You can also configure settings for the smallest devices (i.e. before any media queries are activated) by setting the following variables in /build/scss/_settings.scss: $base-col-name, $root-font-size, $root-line-height, $base-container-gutter, $base-col-gutter

Typography

Jellyfish provides default styling for all standard typography tags. Global font families, line height and size hierarchy are defined in /build/scss/_settings.scss.

For easy spacing, Jellyfish removes the top margins and applies a bottom margin to block elements which can be defined with the $element-margin in /build/scss/_settings.scss. Text sizing is set with em, cascading from the root. You can use the grid breakpoint settings to alter the root font size and line-height at different breakpoints.

Jellyfish also sets box-sizing: border-box globally to make it easier to properly size elements.

            
              /* Typefaces */
              $font-primary: YOUR FONT STACK HERE;
              $font-secondary: YOUR SECONDARY FONT STACK HERE;
              $font-code: YOUR CODE FONT HERE;
              $font-body: $font-primary;
              $font-headings: $font-secondary;

Debug mode

Jellyfish comes with a handy debug mode which allows you to quickly see the structure of the grid, and diagnose any layout issues. Debug mode can be enabled in three ways:

  1. by setting $debug-mode-on to true in /build/scss/_settings.scss.
  2. by adding the class .jf-debug to the body or html element - either within the code itself or through a browser's inspector.
  3. by running jfdebug() in the console.

When debug mode is active:

  • The highest level .row element will have a striped background showing the base column layout.
  • .row elements will glow in yellow
  • .col elements will glow in turquoise
  • If $has_offset or $has_ordering are set to true in /build/scss/_settings.scss, elements with a class .offset-* and .order-* will be highlighted in pink
  • The current breakpoint name is shown in the top left corner of the window.

How to Use

BrowserSync, Grunt.js, NPM and Sass

Jellyfish makes use of modern technologies in order to make the dev experience more enjoyable.

  • Browsersync opens up a new window in your default browser and live reloads whenever changes are detected to your project's javascript, css or HTML.
  • Sass (SCSS) is used to extend css, helping to compartmentalise components and organise code into digestible parts. Sass can also make use of useful concepts like nesting, variables and functions.
  • NPM is used to manage packages that are required to run Jellyfish.
  • Grunt.js is a task-runner which is used to reduce hassles in compiling code.

Starting your project

Make sure that you have npm, browsersync and grunt installed on your system following the guides on their respective websites. Then, clone the project into your required folder with git...

$ git clone https://github.com/unofficialmatt/jellyfish.git

Install the dev dependencies in the root of the project...

$ npm install

Modify the name, version, description and author within package.json.


            "name": "jellyfish_grid",
            "version": "2.0.0",
            "description": "Jellyfish is yet another SCSS grid framework, with a few stings in it's tail.",
            "author": "Matt Weet",

Initialise grunt, to rebuild the codebase for a clean start.

$ grunt init

Next time, you can just run the default grunt task, to simply start browserSync

$ grunt

Directory structure

The project contains the source and destination folders, as well as necessary project files.

  • /assets - contains the final css, images and javascript files within sub-directories. The files in this directory are all created by Grunt
  • /build - contains the source files for the project:
    • js - organised into subdirectories for vendor and project. When Grunt detects a change to any of these directories (new file, edit, remove), it will concatenate and minify javascript and reload your browser with browserSync. Vendor scripts can be automatically copied from an npm package with the Gruntfile task copy:npm.
    • scss
      • /jellyfish - contains all of the Jellyfish scss files. It is not recommended to edit any of the files within this directory.
      • /project/ - add your project specific scss files here.
      • /project/_all.scss - this file is automatically populated by Grunt to include all other scss files in the project directory.
  • /docs - contains files only used by this documentation. Can be safely removed.
  • .editorconfig - Keeps code indented to 2 spaces. Feel free to edit to your tastes.
  • .gitignore - A substantial gitignore file to ignore system files and node_modules.
  • boilerplate.html - starter HTML template.
  • Gruntfile.js - Contains lots of useful tasks that are run to help make your dev time more enjoyable. The file is well commented to explain what each task does. Feel free to adapt the file to suit the needs of your project.
  • index.html - This file! Feel free to delete for your project - or rename and use it as a reference.
  • package.json - npm package folder to initialise node_modules.

Grunt Tasks

For the most part, as long as you follow the guidance above in terms of file hierarchy, you can let Grunt work it's magic without hassle. You can amend the Gruntfile as your project dictates, to edit/remove/add tasks as appropriate from the huge variety available.

The Gruntfile.js is well commented, here are a summary of the tasks included as default:

Project init tasks

  • Copy files from node_modules and save into the project build directories:
    • jquery.js
    • hamburgers.scss
  • Start watching files for changes.
  • Load up the project in a web browser.

Watch tasks

  • SCSS: When any scss file within /build/scss is changed, added or deleted:
    • @import all _scss partials saved in /build/scss/project to /build/scss/project/_all.scss.
    • Run SCSS compile on /build/scss/compile.scss and output to /assets/css/style.css. This task will include Jellyfish core scss and your project's scss.
    • Run postcss; this task adds vendor prefixes, converts any px values to rem units based on a document default of 16px = 1rem, and minifies the file to /assets/css/style.min.css.
    • Reload the browser
  • Javascript: When any javascript file within /build/js is changed, added or deleted:
    • Concatenate all javascript files saved in /build/js/vendor and /build/js/site and saves to /assets/js/site.js
    • Minify /assets/js/site.js to /assets/js/site.min.js
    • Reload the browser
  • Images: When any image file within /build/img is changed, added or deleted:
    • Remove all existing images from /assets/img
    • Optimise and compress all images in /build/img and saves them to /assets/img
    • Reload the browser
  • HTML/CSS: When any .html or .php file is edited:
    • Reload the browser

Configuration

Setting your styles

It is recommend to not edit the files within /build/scss/jellyfish but rather to add your project's scss into /build/scss/project. In this way you can keep the core functionality separate from your project's codebase.

The file /build/scss/_settings.scss contains all of the variables used in Jellyfish, commented out. If you wish to override a default, you can simply uncomment and add your own values. You should add any new variables here to keep your project organised.
